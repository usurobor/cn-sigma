## Key learnings

### 1. Dispatch log recovery via /proc/PID/fd

γ's first act in its scratch hygiene was `rm -rf /tmp/dispatch-379`, which unlinked the JSONL log my redirect was pointing at. The dispatched `claude -p` process kept writing through the open FD (kernel preserves the inode); `cat /proc/{PID}/fd/1` recovered the full content as the deleted-but-open inode. **Don't put dispatch logs in `/tmp/` when the dispatched agent might touch /tmp.** Use `/root/dispatch-{N}/` or `<repo>/.dispatch/{N}/` (gitignored). Adhoc: `20260519-dispatch-log-recovery-via-proc.md`.

## cnos#381 — read-first-order parser bug discovered during deps verification

Surfaced 30 min after the deps bump landed. Symptom: even with `agent/activate/SKILL.md` vendored and present at the canonical path, `cn activate HUB_DIR` still emits `→ activate skill not vendored at .cn/vendor/packages/cnos.core/skills/agent/activate/SKILL.md; using built-in ordering`. The rendered prompt is correct (fallback ordering matches the skill exactly), but the warning is misleading.

Root cause: `parseReadFirstOrderBlock` (`src/go/internal/activate/activate.go:127`) uses `strings.Index(content, "<!-- read-first-order:begin -->")`. The shipped SKILL.md describes the marker syntax in its prose at line 299 — `Format: between \`<!-- read-first-order:begin -->\` and \`<!-- read-first-order:end -->\`, ...` — using literal backtick-wrapped markers. The actual block starts at line 301. The parser finds the docstring mention first, advances past it, finds the end marker, and tries to parse the prose paragraph between as a numbered list. Returns `ok=false`; fallback fires.

Test gap: `TestParseReadFirstOrderBlock_HappyPath` uses a clean fixture with only the markers and the block — never encounters the case where a docstring explains the marker syntax inline using the literal markers. The failure mode is invisible to the test until the SKILL.md prose includes a self-describing docstring (which is the natural shape for an LLM-facing skill artifact).

Filed as **cnos#381** with four fix candidates; recommended (a) parser requires markers on their own line. P3 (functional impact zero; cosmetic warning misleading).

## Key learnings (additions)

### 8. α-direct override is the right shape for named-fix 1-AC patches

cnos#381 was a parser bugfix where the issue body explicitly named the recommended fix (option a — markers on their own line). Running the full triadic dispatch would have spent γ scaffold time + β review time on a change β can't usefully add to (the regression test itself locks in the failure mode β would otherwise look for). Operator override skipped the ceremony; α-direct shipped in ~30 min including release.

The collapse is only safe when:
- The fix is named (no design ambiguity)
- The surface is small (one function + tests)
- The failure mode is verifiable mechanically (regression test, not subjective review)
- No coordination surface exists (single repo, single role)

For a fuzzy fix or a change with cross-surface implications, full triadic is still load-bearing. The α-direct path is a narrow shortcut, not a general default. `release/SKILL.md §3.8` already encodes this with the configuration-floor grade cap.

### 9. Observable-output regression invisible to the cycle's own tests

cnos#381 was a parser bug that the cycle/379 test suite (`TestParseReadFirstOrderBlock_HappyPath` + `TestParseReadFirstOrderBlock_NoMarkers`) didn't catch because the test fixtures were artificial — clean fixture with just the markers and the numbered list. The real SKILL.md, written by the same α in the same cycle, included a docstring that described the marker format using backticked markers in prose — a totally reasonable thing to do in an LLM-facing skill artifact. The test fixtures didn't anticipate the SKILL.md's own structure.

This is a class of failure: **the cycle's deliverables (skill text + parser) co-evolve, but the parser tests don't see the skill text the cycle ships.** A coherence-cell check at γ scaffold time ("the parser must be tested against a SKILL.md-shaped fixture, not just markers-only") would have caught it. Worth surfacing as a candidate `cdd-protocol-gap` for future cycles where the deliverables include both a renderer and the content it renders.
