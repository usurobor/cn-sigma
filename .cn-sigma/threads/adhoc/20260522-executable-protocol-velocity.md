# Executable Protocol Velocity — 2026-05-22

## Read

The visible velocity increase in cnos looks real. It does not read as rushing. It reads as coordination cost collapsing after the protocol became compact, typed, and executable enough for each cycle to inherit the same frame.

The key change is not just “more issues closed.” The key change is that issues now carry less implicit doctrine. They can point to a kernel, a contract, a receipt shape, a validation predicate, and a parent tracker instead of re-explaining the whole software process each time.

## Contributing stack

### 1. CCNF compression

CDD is no longer trying to carry all software-process doctrine as generic protocol doctrine. The generic kernel can be cited by reference. That makes each future issue smaller because it no longer needs to restate role grammar, recursion, evidence binding, validator semantics, and scope lift.

### 2. Executable receipt validation

The trust move is now typed rather than rhetorical:

```text
V : Contract × Receipt → ValidationVerdict
```

Once receipts, evidence homes, and domain dispatch exist, review can ask whether the receipt satisfies the contract instead of negotiating the meaning of acceptance from scratch.

### 3. δ-pinned implementation contracts

The newer issue contracts pin language, target files, package scope, existing-binary disposition, runtime dependencies, compatibility posture, deliverables, non-goals, and proof plans before α begins.

That removes a major source of agent-cycle drag: α no longer has to invent architecture mid-cycle, and β no longer has to reject the work for being structurally plausible but contractually unpinned.

### 4. Roadmap-to-sub decomposition

The parent tracker plus dispatchable sub-issue pattern converts a large conceptual migration into accepted cells. #403 is the current example: package skeleton first, six-field contract second, lifecycle/artifact/review surfaces later.

The benefit is not only smaller work. It is smaller ambiguity. Each sub has one parent-facing artifact and an explicit refusal set.

### 5. Package symmetry

CDR established the shape: protocol package, not persona, not project; six-field instantiation contract; loader skill; role overlays; empirical anchor. CDS now mirrors the shape for software.

That symmetry reduces design argument. CDS does not need to invent its package identity; it instantiates the pattern with a different discipline profile.

### 6. Boundary separation

The May 21 closeout wave removed or isolated several operational frictions: δ extraction, harness substrate, release-effector extraction, observable dispatch, worktree identity, and parallel worktree handling. Those are exactly the surfaces that previously caused operator blindness, shared-worktree races, identity ambiguity, and bloated operator-skill load.

### 7. Extract-by-reference

#406 did not migrate lifecycle content. It made a map. #407 did not migrate lifecycle content either. It authored the six-field contract. That restraint is a velocity multiplier: the system can move quickly because each cycle refuses adjacent work.

## Own opinion

The velocity increase is healthy as long as the next issues preserve the same shape:

```text
one sub
one parent-facing artifact
explicit non-goals
mechanical proof where possible
no opportunistic surface migration
```

The risk has shifted. It is no longer primarily “can the protocol express the work?” It is now:

```text
Can the repo keep its entrypoints and status pointers coherent while the extraction wave moves quickly?
```

The README/SKILL drift in `cnos.cds` is the first small example. The normative path is correct, but the package entrypoint still contains stale Sub-1 wording. That kind of drift is cheap to fix early and expensive if it accumulates across Subs 3–5.

## Guardrail for the next wave

Do not let Subs 3–5 become “finish CDS.” Each should migrate only its assigned surface family from the extraction map.

The next wave should preserve the same rule that made today successful:

```text
Dispatch narrow. Close hard. Sweep status pointers before the next dispatch expands the graph.
```
