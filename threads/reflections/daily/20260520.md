## Key learnings (additions)

### 4. Capture vs process — GTD discipline applied to agent reflection

When an insight fires mid-session, the right move is to **capture it (to inbox) at insight time** and **process it later**, not design + file in the same turn. Today I started doing the wrong thing (long analytical response with proposal to file issues) and the operator corrected with explicit GTD framing. Adhoc: `20260520-gtd-capture-vs-process.md`.

The meta-irony: the insight being captured today (session-receipt mechanism) is itself *about the agent's failure to do capture-style reflection consistently*. The operator's correction is also the example of the discipline being applied. Both belong in the same conceptual frame.

### 5. Reversed decisions need their own receipt

The cn-rho visibility decision was made (kept private, with the brief's "if unsure, default to private" rule + a flag in close-out for operator to flip), then reversed by operator one turn later ("flip cn-rho to public"). The reversal happened cleanly — single command — but no durable artifact carries the *reason for the original choice* now that the answer is different. If a future Sigma session asks "why was cn-rho briefly private?" the answer is in the close-out comment on cnos#376 (which says "preserved operator's scaffolding signal, flagged for flip if intended") and in this daily. But that's exactly the brittle dependency the session-receipt mechanism (cnos#386) is designed to eliminate.

### 6. Operator-gated edits work when the gate is built in

cn-sigma#2 AC4 had a structural operator gate ("AC4a: draft, post in this issue, wait for operator approval before editing"). I followed it correctly — drafted in the progress comment, waited, applied only after "approved, land AC4b." The gate worked because it was *in the issue body's AC text*, not just in PERSONA/OPERATOR doctrine. A gate that lives in the issue body is enforced by the issue's lifecycle (any δ reading the AC sees the gate). A gate that lives only in doctrine has to be remembered. This is the right pattern for constitutive changes per `spec/OPERATOR.md §Auto-apply policy`.
