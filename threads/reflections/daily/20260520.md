## Near-miss — cnos working tree pollution

To read the three staged bundles, ran `git checkout cb1bf430 -- .cdd/iterations/cross-repo/` from `/root/cnos`. This restored content from the source branch into the working tree **AND staged it in the index** (checkout from a ref stages by default). The cnos main HEAD was `5e8cafdb`; the staged bundles + several unrelated bundles from `cb1bf430` were now ready to commit onto cnos main with one `git commit -am`.

I never almost committed it — the next step was reading the files, not committing — but the failure mode is real: a δ working from `/root/cnos` who reaches for `git checkout <ref> --` to read source content can accidentally promote a source-branch's work product onto main. Caught and unstaged before any cross-talk; adhoc filed: `20260520-read-source-branch-without-staging.md`. The durable lesson: for read-only access to a branch's content, use `git show <ref>:<path>` (writes to stdout, doesn't touch the index) or a separate worktree.
