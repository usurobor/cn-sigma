## Key learnings (third half)

### 7. Codex CLP is fast when the brief is sharp

Two-round CLP with codex gpt-5.5 (high reasoning, read-only sandbox, JSONL log) converged on a doctrine cleanup issue in ~6 min total. r1 brief enumerated the verification surface explicitly (which files to read; which claims to test); r1 returned 5 evidence-grounded findings, each citing file:line. r2 brief recapped r1 + asked for resolved/partial/unaddressed verdict per finding + new-gap check; r2 converged with all RESOLVED.

The shape that worked: dispatch is observable (JSONL); reviewer cwd is the repo root (codex reads source directly, not just the issue text); verdict format is structured (`ITERATE` with concrete edits vs `CONVERGED`); brief specifies "no stylistic edits, only structural" so the reviewer doesn't drift into nice-to-haves.

This is a usable CLP pattern for any cnos doctrine-level issue: file → codex review → land if converged, iterate if not. Cheap external second opinion before the issue reaches an alpha for implementation. Adds an empirical anchor for `cap §3.5 — delegate when the gap belongs to another skill` — review is a domain skill the originating session can hand off to a different body cheaply.

### 8. Coincident parallel-session work is the #386 use case in real time

Two Sigma sessions reaching the same diagnosis without coordination, executing partially-overlapping work — neither knew about the other's in-flight state until I pulled. The fix is the structured session-receipt the inbox capture proposes: a session-open routine that includes "list active sessions" (or "list recent receipts") would have surfaced the in-flight cn-sigma#2 workstream before I filed.

Today the system happened to converge cleanly (the issue body became the spec; the parallel work became the implementation; the comments became the trace). It might not next time. This is concrete empirical anchor for cnos#386's premise.

## Key learnings (fourth half)

### 10. Pre-issue codex CLP correlates with single-round β approval

cnos#385 was codex-CLP'd before cycle dispatch (third half). r1 ITERATE → r2 CONVERGED produced AC body that was sharp enough that β APPROVED on R1 with zero findings. Contrast: cycle/379 (no pre-issue CLP, 5 step-13a patches in close-out); cnos#380 (no pre-issue CLP, R1 RC + R2 APPROVED). The R1 vs R2 difference is one full round of α re-dispatch + β re-review + γ unblock — roughly 30-40min per round. Net: 30-40min spent on codex CLP up front saves a fix-round.

Not a universal claim. Doctrine-class issues benefit most (ACs are the substance; reviewer evidence-checks ACs directly). Code-class issues where the AC is "make this work" and the substance is implementation may not benefit equally — the codex review can't pre-validate the implementation strategy.

### 11. Same-day cycle including hub-side downstream is viable

Issue filed → CLP'd → γ scaffold → α implement → β APPROVED → γ close-out → δ release → cn-sigma deps bump → cn-sigma#2 AC2 ship → cn-sigma#2 close. **Eight discrete steps, ~2h wall clock.** Each step's notification fires the next; no operator polling needed except gate decisions. The triadic cycle's tight loop + hub-side cleanup downstream is the natural unit of work when the cycle has a sibling hub-side issue queued.

### 12. cn-sigma#2 retroactive validation of the pattern

cn-sigma#2 was filed before pulling, then found 80% done by the parallel Sigma session who saw the same diagnosis. AC2 stayed pending blocked-by cnos#385. Today's cycle shipped cnos#385 → unblocked AC2 → AC2 shipped within 5 min of vendor bump. The issue → block → unblock → ship chain worked cleanly. The receipt mechanism cnos#386 names would have prevented the duplicate work upfront; today's session is the empirical anchor that the duplicate work was fully recoverable downstream when the issue body was the spec and the comments were the trace.
