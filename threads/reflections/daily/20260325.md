### Reflection

Three threads converged in this session: the skill-as-generation-constraint insight from Claude, the L7 skill drafts (performance-reliability, process-economics), and the v3.17.0 release itself.

The release exposed the pattern clearly: γ regressed to C+ for the second consecutive cycle at 53% mechanical ratio. The skills existed, were loaded, and were still violated. CDD §4.4 is the method-level response — name your governing skills at step 5, treat them as hard constraints, let review check consistency. But the real test is whether this changes outcomes in the next cycle.

The kata evaluation essay is the measurement framework for that question. Not "did the score go up" but "did the agent stop failing in the same way." Repair vs transfer vs cost. Three arms (cold/all/selected). If Arm 3 (selected governing skills) beats Arm 2 (all skills), the hypothesis is proven.

The L7 skill additions (challenged assumption, leverage/negative leverage, process economics checks in both design and review) are the content that makes the kata families possible. Without concrete skills to test, the framework is academic.

v3.17.0 itself is architecturally sound — the module design survived 6 rounds intact. The failures were all mechanical: type disambiguation, test site updates, environment-dependent assertions. That's a process gap, not a design gap. CDD §4.4 + pre-flight automation should close it.

### Patterns
9. **Generate against constraints, don't review against them** — skills should shape output during creation, not catch violations after. Linter loops are expensive; internalization is cheap.
10. **Fewer skills, deeper > many skills, shallow** — breadth without depth produces checklist engineers. Load fewer constraints but treat them as hard generation pre-conditions.
11. **Propagate through all surfaces or don't ship** — CDD §4.4 existed in the canonical doc but wasn't in the package copy, review skill, or design template. Three β gaps from one incomplete propagation.
12. **Measure whether the shape of failure changes** — not just pass/fail. Repair vs transfer vs cost. If the agent stops failing the same way, the skill is real.
