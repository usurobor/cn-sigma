# agent/dispatch-protocol: label-gated issue queue for implementation cells
Labels: kind/skill, priority/P1, area/agent

Priority: P1 — control-plane prerequisite for cnos#450 + cnos#444 implementation cycles. Without it, those implementations risk re-inventing incompatible dispatch behavior.
Status: New protocol/skill. Defines the minimal GitHub issue-label discipline by which a human authorizes a wake worker to claim and execute one implementation cell.

## Purpose

Define the minimal GitHub issue-label protocol by which a human authorizes a wake worker to claim and execute one implementation cell. The protocol is canonical (no per-org overrides in v0).

## Core invariant

An issue is **executable** only when ALL of:
- open
- carries `dispatch:cell` label
- carries `status:todo` label

The wake claims **at most one** executable issue per run. Lifecycle (`status:*`) ≠ issue type (`dispatch:cell`).

## Rationale for the dispatch:cell + status:todo composition

`status:todo` alone is not enough. Master issues, RFC review issues, meta-planning issues, and implementation cells can all be "todo" in a human board sense. Only **implementation-cell** issues should be executable. `dispatch:cell` is the type marker; `status:todo` is the authorization gate. Both required.

## Acceptance criteria

### AC1: canonical labels defined

Labels (v0, no per-org override):

```
status:backlog
status:ready
status:todo
status:in-progress
status:review
status:changes
status:blocked
dispatch:cell
kind/wave        (optional, recommended for master issues)
kind/rfc         (optional, for RFC/review issues)
```

Rules:
- Exactly one `status:*` label on an open issue.
- Only `status:todo` can dispatch.
- Only `dispatch:cell` + `status:todo` can be claimed.
- Closed issue = done.
- No `status:done` in v0.

### AC2: human dispatch gate

- `status:ready` means spec'd but not authorized.
- `status:todo` means human-authorized dispatch.
- Planner (interactive Sigma) may apply `status:todo` only when quoting an explicit human approval comment.
- Otherwise human applies it directly.

### AC3: dispatchable marker required

Wake only claims issues with `dispatch:cell` + `status:todo`. Master issues, RFCs, review issues, and planning issues MUST NOT carry `dispatch:cell`.

### AC4: claim operation defined

Claim sequence:
```
1. Wake receives issue-labeled event or scheduled sweep.
2. Re-read issue labels (do not trust event payload alone).
3. Verify:
   - issue open
   - dispatch:cell present
   - status:todo present
   - status:in-progress absent
   - status:blocked absent
   - status:review absent
4. Remove status:todo label.
5. Add status:in-progress label.
6. Write claim comment:
   claimed_by: <hub>/<run-id>
   claimed_at: <timestamp>
   head: <commit-sha>
7. Re-read and verify status:in-progress is present.
8. Only then launch the cell.
```

### AC5: no double execution

GitHub labels are not a true compare-and-swap primitive. Anti-double-execution discipline composes:

- Repository-level workflow `concurrency:` group (GH Actions serialization).
- Claim-time re-read + label transition.
- Claim comment with run id.
- Scheduled sweep is backstop for missed/canceled events.

Wake template uses one repository-level concurrency group for dispatching. Scheduled sweep claims at most one issue per run. Issue-labeled event claims only the event issue. Schedule remains backstop.

### AC6: cell boundary defined

The dispatch protocol does NOT implement the cell. It only selects and claims work. The claimed issue is passed to the CDD cell runtime:
```
γ coordinate
α implement
β review
γ receipt
```

`cohere` (cnos#444) = dispatcher/orchestrator. Cell = claimed-issue implementation runtime. γ/α/β run inside the cell, scoped to the claimed issue.

### AC7: review / rejection / blocked transitions

```
success:
  status:in-progress → status:review
  with PR/receipt
human accepts:
  close issue
human rejects:
  status:review → status:changes
  with comment
planner/human repairs:
  status:changes → status:ready or status:todo
blocked:
  remove status:in-progress
  add status:blocked
  comment names required human/operator input
```

### AC8: wave semantics

- Master issue tracks dependency map; carries `kind/wave`; MUST NOT carry `dispatch:cell`.
- Each subissue is independently dispatchable only when it has both `dispatch:cell` and `status:todo`.

### AC9: drift handling

If an issue has multiple `status:*` labels, no status label, or illegal status combination:
- Do not claim.
- Report `degraded_reason: dispatch_label_drift` (extend AC14a canonical list at cnos#444).
- Comment with repair instruction.

### AC10: template integration

cnos#450 wake-template cites this protocol and renders:
- `on: issues` labeled trigger
- scheduled sweep trigger
- concurrency group
- dispatch query
- claim/report steps

### AC11: priority deferred

No priority labels in v0. If dependencies matter, do NOT label downstream subissues `status:todo` yet. Deterministic claim order on scheduled sweep: lowest issue number among `dispatch:cell` + `status:todo`.

## Scope

In scope:
- Canonical label set (AC1)
- Claim operation + transitions (AC4, AC7)
- Concurrency discipline (AC5)
- Wave semantics (AC8)
- Drift handling (AC9)
- Citations from cnos#450 + cnos#444 (AC10)
