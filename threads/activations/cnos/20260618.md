Labels: kind/skill, priority/P1, area/agent

Status: New skill + contract. Defines the external-scheduler → wake bridge so cron source can be moved off best-effort GitHub Actions to a reliable substrate.

## Problem

What exists:
- Each hub's wake yaml uses GitHub Actions `schedule:` cron with 4 staggered minutes per the v1 mitigation (cnos `c619a14300`, 2026-06-02). GitHub docs explicitly note cron is best-effort with high drop rate during cluster load.
- Original RCA (cnos `356797c3`, 2026-06-02T20:35Z) identified three escalation options: (1) multi-cron stagger — done; (2) external scheduler via repository_dispatch — pending; (3) self-rescheduling sleep + workflow_dispatch — hacky.
- Empirical evidence as of 2026-06-18: cnos wake silent for ~10h despite 4 cron slots. Multi-cron is failing at scale.

What is expected:
- A reliable external scheduler (Cloudflare Worker, Vercel cron, AWS EventBridge, or equivalent) POSTs `repository_dispatch` events to each hub's GitHub API.
- Each hub's wake yaml listens on `repository_dispatch:` event_type `wake`, AND keeps existing `schedule:` cron as backup.
- One PAT (operator-owned) lives in the external scheduler. No PATs distributed to hubs.

- cron sources at GH Actions are not reliable enough to support the cohere wave or any production rollout.
- The wave's reviewer-paused state is partly amplified by cron drops (wake can't pick up directives when cron drops slots).

- Without this, the wave's progress is gated by GH Actions cron's drop rate. Productivity scales inversely with body count.
