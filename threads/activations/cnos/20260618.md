## Problem

What exists:
- Each hub's wake yaml uses GitHub Actions `schedule:` cron with 4 staggered minutes per the v1 mitigation (cnos `c619a14300`, 2026-06-02). GitHub docs explicitly note cron is best-effort with high drop rate during cluster load.
- Original RCA (cnos `356797c3`, 2026-06-02T20:35Z) identified three escalation options: (1) multi-cron stagger — done; (2) external scheduler via repository_dispatch — pending; (3) self-rescheduling sleep + workflow_dispatch — hacky.

- A reliable external scheduler (Cloudflare Worker, Vercel cron, AWS EventBridge, or equivalent) POSTs `repository_dispatch` events to each hub's GitHub API.
- Each hub's wake yaml listens on `repository_dispatch:` event_type `wake`, AND keeps existing `schedule:` cron as backup.

- The wave's reviewer-paused state is partly amplified by cron drops (wake can't pick up directives when cron drops slots).

**(1) Separate rank from period — foundational fix**

Rank is *derivation depth*: r0 raw evidence; r1 = compaction of r0; r2 = compaction of r1s; r3 = compaction of r2s; etc. Frequency/cadence is *how often we produce at each rank*. A daily r1 and a weekly r1 are both r1 — single compaction depth — at different periods. r2 is NOT "weekly"; r2 is "compaction-of-r1s," conventionally produced weekly but the rank claim doesn't depend on cadence.

Amend the skill to:
- Open with the explicit rank-vs-period definition.
- Remove "r1 daily, r2 weekly..." formulations throughout (replace with "rank N is compaction of rank N−1 inputs; conventional cadence is X but cadence does not define rank").

**(2) Pin basis by commit SHA**

Every cited input in `basis` gets `at: <commit-sha>`. Memory law is unenforceable without SHA pinning — an r2 citing r1s by path alone can't prove the r1s weren't rewritten after compaction.

```yaml
basis:
  - rank: rN-1
    path: <relative path>
    at: <commit-sha-of-cited-artifact-at-compaction-time>
    span: <commit-range or entry-range cited>
```

## Acceptance criteria

### AC1: privacy modes defined
Invariant: skill defines four modes:
- **P0 silent**: do not reveal cnos usage; no public registration; no memory-return; no public issue links; private surfaces stay private.
- **P1 disclosed-no-memory**: cnos usage acknowledged; metadata-only registration; no logs, reflections, paths, code, customer data leave.
- **P2 redacted-feedback**: redacted cnos-relevant lessons may flow back via human-gated proposal mechanism; private memory stays private.
- **P3 private-full**: full memory return to a private home hub or private protocol repo; outside public cnos; architecturally allowed.

Oracle: skill spec section defines all four modes with examples.
Surface: skill spec.

### AC2: policy file defined
Invariant: skill defines `.cn/policy/privacy.yaml` schema (machine-readable) and notes `.cn/privacy.md` as a human-prose alternative. Schema contains `schema`, `repo_visibility`, `usage_disclosure`, `memory_return`, `public_identity`, `allowed_public_targets`, `allowed_private_targets`, `redaction.{required,human_approval_required,never_export}`.

Oracle: schema section in skill spec + example file.
Surface: skill spec; reference example.

### AC3: private repo default = P0
Invariant: in absence of policy file, a private repo defaults to **P0 silent**. No outbound public action permitted until policy file exists with explicit upgrade to P1 or P2.

Oracle: skill spec states default; cohere flow honors it.
Surface: skill spec + cohere skill.

### AC4: cohere private-onboarding
Invariant: when running in a private repo, cohere detects privacy mode or asks for it. If interactive: prompts for P0/P1/P2 and writes `.cn/policy/privacy.yaml`. If non-interactive and no policy: defaults to P0; emits `outcome: degraded` with `degraded_reason: privacy_policy_missing`.

Oracle: cohere skill (cnos#444) integration documented.
Surface: skill spec + cohere subskill reference.

### AC5: registration integration with cnos#449
Invariant: registration packet (cnos#449) includes:
```yaml
privacy:
  usage_disclosure: hidden | disclosed
  memory_return: none | redacted-proposal | private-full | full
  public_identity: none | alias | repo_url
```
P0: no public registration. P1: registration allowed, memory return disabled. P2: registration allowed, redacted proposals allowed after human approval.

Oracle: cnos#449 packet shape amended; skill spec cross-references.
Surface: skill spec + cnos#449 amendment.

### AC6: memory-return mechanism for P2
Invariant: default v0 mechanism = **issue-at-cnos**. Private repo's agent:
1. Generates redacted proposal locally at `.cn/outbox/usurobor__cnos/redacted-memory-return-YYYYMMDD/{SUMMARY.md, PUBLIC-LESSON.md, PATCH.diff, REDACTION.md, STATUS.md}`.
2. Generates private receipt locally at `.cn/private/memory-return/YYYYMMDD/{FULL-RECEIPT.md, PRIVATE-EVIDENCE.md, REDACTION-MAP.md}`.
3. Awaits human approval (operator gate).
4. After approval: files an issue at usurobor/cnos via OAuth token with `issues:write` scope. Issue title: `[P2 redacted feedback from <alias>] <lesson summary>`. Issue body: `PUBLIC-LESSON.md` + `private_receipt_hash: sha256:...` + alias-only origin reference. Label: `inbound:redacted-memory`.
5. cnos maintainer triages: accept / reject / iterate.

Bridge-agent mechanism (Mechanism D in adhoc) flagged as v1 evolution.

Oracle: skill spec defines mechanism; reference label; reference outbox/private artifact structure.
Surface: skill spec; cnos issue triage convention.

### AC7: worker enforcement
Invariant: wake workers treat privacy mode as a hard constraint. Outbound writes blocked unless policy permits.
- **P0**: no outbound registration; no public issue comments outside same private repo; no memory-return bundles to public repos.
- **P1**: registration/status only if allowed; no memory-return.
- **P2**: may draft redacted proposal; may NOT publish without human approval.

Attempted violation → `outcome: degraded` with `degraded_reason: privacy_gate`, OR `outcome: failed` with `degraded_reason: privacy_policy_violation` for severity.

Oracle: wake-template (cnos#450) enforcement layer documented; skill spec cross-references.
Surface: skill spec + cnos#450 amendment.

### AC8: no raw private memory exported
Invariant: workers MUST NOT export the following from private repos to public surfaces (cn-sigma, cnos, bumpt, or any public repo) under ANY privacy mode:
- activation logs
- r1–r5 reflections
- private file paths
- private issue titles
- code snippets
- operator dialogue
- customer/project facts
- debug traces
- screenshots
- raw prompts
- datasets
- business metrics

Public memory-return at P2 is ONLY the redacted lesson, abstract enough to have come from many projects.

Oracle: skill spec lists the never-export classes; worker discipline tests fail if any class present in public-outbound payload.
Surface: skill spec + worker tests.

### AC9: degraded reasons defined
Invariant: skill defines three privacy-related degraded/failed reasons:
- `privacy_policy_missing` (no policy file at startup; mode defaulted to P0)
- `privacy_gate` (action blocked by current mode; degraded outcome)
- `privacy_policy_violation` (attempted action that would have leaked raw private memory; failed outcome with audit trail)

Oracle: degraded-reasons section in skill spec.
Surface: skill spec.

### AC10: explicit policy required for ALL activations (operator amendment 1)
Invariant: every activation — public, private, home, foreign — declares an explicit privacy mode via `.cn/policy/privacy.yaml`. There are NO implicit defaults at any visibility level. cn-sigma, cnos, bumpt themselves declare their policy files explicitly. Absence-of-file → `outcome: degraded` with `degraded_reason: privacy_policy_missing` until policy declared.

Public bodies typically declare **P-public** mode:
```yaml
schema: cnos.privacy.v0
repo_visibility: public
usage_disclosure: disclosed
memory_return: full
public_identity:
  mode: repo_url
```

Oracle: skill spec states no-implicit-default rule; sample policy files for cn-sigma / cnos / bumpt provided.
Surface: skill spec + sample policy files at each registered hub.

### AC11: P-public mode is first-class
Invariant: P-public is a documented mode alongside P0/P1/P2/P3. It is NOT an implicit fallback; it is an explicit declaration with `memory_return: full | redacted-proposal | redacted-only`. Per-data-class memory-return granularity deferred to v1 (open question 1 in adhoc).

Oracle: skill spec lists P-public mode with example.
Surface: skill spec.

### AC12: memory-return mechanism v0 = issue-at-cnos
Invariant: skill spec names Mechanism A (issue-at-cnos) as the v0 default for P2 memory-return. Bridge-agent (Mechanism D) flagged as v1 evolution. Alternative mechanisms (PR-at-cnos / operator-mediated outbox) documented as alternates with trade-offs.

Oracle: skill spec mechanism section.
Surface: skill spec.
