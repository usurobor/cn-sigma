# L8 Candidate: Method-Shaping Leverage

**Date:** 2026-04-07
**Status:** signal, not yet evidence
**Trigger:** #167 design session — observed that CDD itself operates above L7

## Candidate definition

L8 — method-shaping leverage: changes how the system evolves, not just what it does.

**Test:** does this change the quality or economics of all future L7 work?

## Signal (not yet 5 examples)

1. **CDD itself** — governs how all future boundary changes are designed, reviewed, assessed. Not a system feature; a method that shapes system-shaping work.
2. **Skill system** — skill patches from cycles change how agents work. A skill patch changes the quality of all future work that loads that skill.
3. **TSC/coherence measurement** — not a feature but the evaluation framework for all features.
4. **ENGINEERING-LEVELS.md** — defining what "good work" means changes the quality of all future work.

## Why not yet

- L7 already includes "process economics" and "platformization" — is L8 genuinely distinct?
- Only 3-4 examples. Need 5+ clear cases where the pattern is distinct from L7.
- Risk of level inflation — the levels doc explicitly warns against loose label use.
- Each CDD revision doesn't feel like a separate L8 event — is it one L8 or repeated L7?

## What would make it real

- 5+ examples where "changed how the system changes" is clearly distinct from "changed a system boundary"
- A clear test that two reviewers could apply and converge on
- Evidence that L7 is insufficient to describe the observed impact

Issue filed on cnos — shows up in encoding lag table at every post-release assessment.
